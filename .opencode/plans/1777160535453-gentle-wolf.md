# Plan: Publish Patchlane to npm

## Goal

Publish patchlane as an npm package so consumers can run it via `npx patchlane` instead of cloning and building the repository inside GitHub Actions workflows.

## Key Design Decisions

- **CLI tool**: `cac@7.0.0` (minimal, zero dependencies, supports both CLI flags and env var fallbacks)
- **Interface**: Hybrid — CLI flags take precedence over env vars. Existing scripts continue to work unchanged when executed directly.
- **Backward compatibility**: Existing reusable workflows remain functional but are updated to use `npx patchlane` internally.
- **Automated publishing**: New workflow triggers on GitHub Release creation.

## Files to Modify

### 1. `src/integration-sync.ts`

- Extract the body of `main()` into a new exported async function `runIntegrationSync(options: IntegrationSyncOptions)`.
- Define `IntegrationSyncOptions` interface with all currently-env-driven settings.
- Keep `main()` for backward compatibility: it reads env vars, builds the options object, and calls `runIntegrationSync()`.
- Gate the `main()` call at the bottom so it only runs when the file is executed directly (not when imported by the CLI):
  ```ts
  import { fileURLToPath } from "node:url";
  if (process.argv[1] === fileURLToPath(import.meta.url)) {
    main();
  }
  ```

### 2. `src/promote-sync.ts`

- Same refactoring pattern as `integration-sync.ts`:
  - Export `runPromoteSync(options: PromoteSyncOptions)`.
  - Keep `main()` as the env-var wrapper.
  - Gate direct execution.

### 3. `src/cli.ts` (new file)

- Import `cac` and the two `run*Sync` functions.
- Define the CLI:
  ```
  patchlane sync   [--upstream-owner] [--upstream-repo] [--patch-refs] ...
  patchlane promote [--expected-sync-sha] [--base-branch] ...
  ```
- For each flag, fall back to the corresponding environment variable (e.g. `--upstream-owner` -> `UPSTREAM_OWNER`).
- Convert kebab-case flags to the options interface and call the appropriate `run*Sync` function.
- Add a shebang (`#!/usr/bin/env node`).

### 4. `package.json`

- Add `cac` to `dependencies` (not `devDependencies`).
- Add `bin` field:
  ```json
  "bin": {
    "patchlane": "./dist/cli.js"
  }
  ```
- Add `files` field to limit the published tarball:
  ```json
  "files": ["dist/"]
  ```
- Add `engines` field:
  ```json
  "engines": { "node": ">=22" }
  ```
- Add `publishConfig` for public access:
  ```json
  "publishConfig": { "access": "public" }
  ```
- Add scripts:
  ```json
  "prepublishOnly": "npm run build"
  ```

### 5. `.github/workflows/patchlane.yml`

- Remove the `Checkout Patchlane implementation` step entirely.
- Remove `Setup Node`, `Install Patchlane dependencies`, and `Build Patchlane` steps.
- Replace the `Run integration sync` step with:
  ```yaml
  - name: Run integration sync
    id: sync
    run: npx patchlane@latest sync
    env:
      UPSTREAM_OWNER: ${{ inputs.upstream_owner }}
      UPSTREAM_REPO: ${{ inputs.upstream_repo }}
      ... (all existing env vars)
  ```
- Deprecate `implementation_repository` and `implementation_ref` inputs by removing their usage from `env` and steps (keep them in the interface to avoid breaking existing callers, or document the deprecation).

### 6. `.github/workflows/promote.yml`

- Apply the same simplification as `patchlane.yml`:
  - Remove clone/build steps.
  - Use `npx patchlane@latest promote` with env vars.

### 7. `.github/workflows/publish.yml` (new file)

- Trigger: `on: release: types: [created]`
- Steps:
  1. Checkout repo
  2. Setup Node (with `registry-url: https://registry.npmjs.org/`)
  3. `npm ci`
  4. `npm test`
  5. `npm publish`
- Use `secrets.NPM_TOKEN` for authentication.

### 8. `examples/sync-upstream.yml`

- Update to show both:
  - Using the updated reusable workflow (simpler, no `implementation_repository`)
  - Direct CLI usage in a consumer-owned workflow (for power users)

### 9. `examples/promote-tested-sync.yml`

- Same updates as `sync-upstream.yml`.

### 10. `README.md`

- Update **Quick Start** to reference `npx patchlane`.
- Add a new **npm / CLI Usage** section documenting `patchlane sync` and `patchlane promote` flags.
- Update the **Configuration Reference** table to note that inputs can be passed as CLI flags or env vars.
- Document the automated publish workflow.

## Verification Plan

1. **Build and tests pass**: `npm test` must pass after refactoring.
2. **CLI works locally**:
   ```bash
   npm run build
   node dist/cli.js sync --help
   node dist/cli.js promote --help
   ```
3. **Direct execution still works**: `node dist/integration-sync.js` and `node dist/promote-sync.js` with env vars must continue to function (integration tests cover this).
4. **npm pack dry run**: `npm pack --dry-run` shows only `dist/` and auto-included files.
5. **Workflow validation**: After merging, create a GitHub Release to verify the publish workflow publishes to npm.

## Out of Scope

- Renaming or removing the reusable workflows entirely (they are updated in-place).
- Changing the core patch-application logic.
- Adding new features beyond CLI argument parsing.
